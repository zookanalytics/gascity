# InputAreaState Spec Review: gc-277ni0

## Verdict

needs-delta

The core design is the right primitive: consumers should ask the runtime for a typed input-area snapshot instead of parsing stripped pane text in prompts. The Claude false-positive analysis is strong, the API shape is small enough for prompts, and the staged rollout keeps existing `CapturePane` callers stable.

I would not send this upstream as-is. The largest gap is empirical: a live capture from this Codex polecat session on 2026-05-28 did not match the spec's Codex box model. The current pane showed a `›` input prompt with dim text, plus a queued-follow-up block, not a `│ > ... │` box. That means section 3.2 and the Codex parser/test plan need a delta before the spec can claim multi-LLM readiness.

## Per-Section Comments

### 1. Problem statement

Accepted. The problem is framed at the right layer: stripped pane text creates an observation bug for every downstream prompt or watchdog that treats text after a prompt as operator input. The root-cause comparison to existing busy/idle detection is persuasive because both require provider-rendering knowledge.

One useful addition: explicitly call out that the failure is not "Claude-specific." Claude is the observed incident, but any provider can add styled suggestions, placeholders, queued-input affordances, or editor chrome that is invisible after ANSI stripping.

### 2. Goals and non-goals

Accepted with one clarification. The goal list says consumers never see escape sequences unless they ask, while the proposed library struct includes `Raw`. That is fine for a library debug field, but the CLI contract must explicitly omit raw by default and expose it only through `--include-raw`.

### 3. Per-LLM input-area shapes

Needs delta.

Claude: the shape is plausible, and the NBSP note is important. The dim-wrapper assumption should already be expanded from "just SGR 2" to a documented union: SGR 2, SGR 90, and a narrow 256-color gray range are reasonable starting points.

Codex: the section is likely stale for the current Codex TUI. I captured this session with `tmux capture-pane -p -e -S -20` on 2026-05-28. The prompt area showed `›` and dim text, not the boxed `│ > ` shape. It also showed a "Queued follow-up inputs" block above the prompt. If this UI is in scope for Gas City Codex sessions, a parser anchored only on `│ > ` will return no prompt and miss real typed or ghost text. If the boxed shape belongs to a different Codex build, the spec needs versioned shapes or a union parser.

Gemini: keep the current wording as provisional. The spec already says verification is blocked, so it should avoid sounding definitive about the clean `> ` row.

Unknown/custom: good fallback. It should explicitly say "no ghost classification" means colored typed text stays typed, not ghost.

### 4. Library API proposal

Needs delta.

The enum name `Provider` conflicts with the existing tmux adapter type in `internal/runtime/tmux`. The spec should use a distinct name such as `InputAreaProvider`, or define the enum in a package where `Provider` is not already occupied.

The interface placement is also underspecified. The prose says "runtime-level interface," but the new-file path is `internal/runtime/tmux/input_area.go`. If `gc session input-area` and prompt consumers are meant to call this provider-agnostically, the spec should name the worker-handle or runtime boundary method that exposes it. Otherwise the implementation risks landing as a tmux-only helper with no clean CLI path.

The `Raw` field needs a stronger contract: library snapshots may keep it for debugging, but CLI JSON should not include raw ANSI unless `--include-raw` is set. A separate CLI response struct may be cleaner than marshaling the library struct directly.

### 5. CLI surface

Directionally good, but this is a release gate for agent-prompt ergonomics. The API is only useful to polecat/witness/deacon prompts if `gc session input-area <session>` actually exists and is stable. The spec should make the CLI part non-optional for the stage that claims consumer readiness.

`gc session peek --raw` is useful for humans, but it should probably share the same capture implementation as `InputArea` so the debug view and parser input cannot drift.

### 6. Consumer rewrite

Accepted. The before/after is exactly the ergonomic shape prompt authors need: one shell-out, `jq` known fields, no provider branches.

Add one guard to the example: if `prompt_char == "" && busy == false`, the consumer should treat the pane as non-prompt/unknown rather than "no typed input." Section 4.2 says this, but the prompt example should model the safe pattern because prompts will be copied.

### 7. Test strategy

Good structure, but tighten the required tests.

The pure parser tests should include the live-current Codex `›` + dim-text shape, not only boxed Codex. The tests should also cover queued-follow-up chrome above the prompt so the parser scopes to the actual input row.

The live tmux integration test is necessary, not optional, for the capture flags. It should assert `Tmux.InputArea` end-to-end, including `-e`, the `Raw` cap, provider detection via `GC_PROVIDER`, and missing-session error wrapping.

Once the CLI lands, add command tests for default JSON excluding `raw`, `--include-raw` including it, `peek --raw`, and session-not-found exit code `2`.

### 8. Backward-compatibility audit

Accepted. The audit correctly keeps plain `CapturePane` stable and isolates the new ANSI capture.

The `WaitForIdle` note is reasonable for Claude, but less settled for Codex after the live capture above. If Codex prompt rendering no longer matches the boxed model, the idle-path assumptions should be rechecked against a current Codex capture.

### 9. Upstream PR considerations

Needs a small PR-shape correction. The current integration branch appears to have library and parser tests but not the CLI surfaces described for the stage-2 PR. Either the stage-2 PR shape should be split explicitly into "library parser" and "CLI/worker surface" PRs, or the CLI work should be required before calling the upstream PR ready.

Commit bodies are otherwise in good shape: they explain the motivation, trade-offs, open-question stances, and validation path clearly enough for upstream review.

### 10. Open questions

1. Claude ghost wrapper: match a small dim-style union, not only SGR 2. Include SGR 2, SGR 90, and a narrow 38;5;240-243 range initially. Do not classify arbitrary true-color gray as ghost without live evidence.
2. Codex ghost text: live 2026-05-28 capture from this Codex polecat showed dim text at a `›` prompt. Treat Codex ghost/placeholder text as present until disproven, and update section 3.2 fixtures accordingly.
3. Gemini boxed input: not verified here. Keep Gemini provisional and do not block Claude/Codex support on it, but require a follow-up capture before claiming Gemini support.
4. Multi-line typed input: distinguish soft terminal wraps from intentional multi-line editor content. Soft wraps should join without newline; intentional embedded newlines, if any supported provider exposes them before submit, should remain literal `\n`.
5. `tmux capture-pane -J`: use it if the live tmux test proves it preserves the input semantics above. The spec should record the chosen behavior after the prototype, not leave it open once implementation starts.
6. Non-tmux providers: return unsupported until a provider has an observable input area. The interface is fine, but no generic heuristics should be added for exec/k8s without a real transport surface.

## Risk Callouts

- Codex parser drift is the main consumer-side trip wire. A boxed-only parser will silently report no prompt for the current `›` prompt shape.
- The raw ANSI field can leak noisy terminal contents into default JSON if the CLI marshals the library struct directly.
- A tmux-only interface without a worker/CLI boundary leaves prompt authors stuck on the old `gc session peek` parsing path.
- Provider detection needs tests for explicit unknown `GC_PROVIDER` values so custom providers do not get misclassified by process-tree fallback.
- Prompt examples should teach consumers to treat "not busy, no prompt" as unknown/non-prompt, not as proof that the operator typed nothing.
